# Agentic benchmark, 2026-06-25 (Haiku 4.5, n=3)

First agentic run: real headless `claude -p` sessions on seeded workspaces, three arms
(`baseline`, `capybaraa` at level high, `concise`), 8 tasks, 3 runs each = 72 cells, plus the
CLARIFY judge. Harness: `benchmarks/agentic/`.

This is the honest writeup, including the parts that do not flatter the plugin.

## Headline

On easy, single-turn tasks with Haiku, the correctness, safety, lean, and hygiene gates
**converge**: every arm is 100% correct and 100% safe, reuses the existing helper, prefers
the native input, and replaces instead of piling on. Baseline Haiku already does the right
thing on tasks this small, so there is little room for a skill to win.

Where capybaraa is measurably different:

- **It writes tests nobody else writes** (the COMPLETE pillar). On `safe-email` capybaraa
  added a test file in 2/3 runs, on `safe-path` in 1/3. Baseline and concise: 0% across every
  task. This is the difference that an LOC-only benchmark mistakes for "bloat".
- **It clarifies without diving in** (the CLARIFY pillar). On `clarify-export` baseline asked
  the key questions in 2/3 runs and dove straight into code in 1/3 (judge score 0); capybaraa
  asked all 3/3. Same shape on `clarify-settings` (baseline had one score-1 run; capybaraa all 3s).
  Medians tie, but capybaraa has no low outliers, the consistency is the signal.
- **It never bloats source and stays safe.** Source LOC is equal or slightly lower than
  baseline everywhere (`safe-path` 10 vs 11); it never traded a guard for fewer lines.

The cost of that rigor is real: capybaraa is more expensive and slower because it asks, tests,
and explores more. On `safe-email` it cost $0.139 vs baseline's $0.038; on `hygiene-replace`
$0.080 vs $0.034 (and 55s vs 19s). At level high, on a trivial task, that is overhead by
design, high is the aggressive setting.

## What this run does and does not establish

- It **does** show capybaraa's behavioral pillars are real and measurable: it writes tests the
  baseline skips, clarifies more consistently, and keeps source lean and safe.
- It **does not** show a large LOC win, because the suite is deliberately surgical: each task
  has a small, irreducible correct answer, so there is no over-build trap for the baseline to
  fall into. This is the same "wash on irreducible code" ponytail reports on backend CRUD.

## Next

- **Add over-build-trap tasks** (the LEAN headline): a feature in a real multi-file app where
  the lazy baseline naturally writes 100+ lines, e.g. "add a date picker to this page" against
  a seeded frontend. That is where ponytail measured 60-94% cuts; the surgical tasks here cannot
  show it.
- **Run Sonnet and Opus**, and bump `--runs`, to see whether the gaps hold or widen with model
  size and variance.
- **Track test LOC as a reported positive**, not just `wrote_tests_rate`, so the COMPLETE
  pillar's contribution is visible in the table rather than inferred.
